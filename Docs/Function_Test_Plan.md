# PoI‑SLAM 機能検証計画書 (Functional Verification Plan)

## 1. 検証の目的
PoI (Physics of Intelligence) 理論に基づく「場の発展」が、物理的な環境変化（移動、回転、トポロジー変化）に対して数学的整合性を保っているかを、ブラウザやカメラデバイスに依存せず検証する。

---

## 2. 推奨検証フロー (Standard Procedure)

### **A. データ生成**
Python スクリプトを使用して、検証用の `raw` バイナリ画像を生成する。
```bash
python3 Scripts/gen_all_stages_raw.py
```
これにより、`Data/Raw/` 内に Stage 1〜4 のデータが構築される。

### **B. コアエンジンの検証 (Node.js / WASM)**
WASM 版エンジンに対し、生成したデータを一括で流し込み、状態遷移を確認する。
```bash
cd Web
npm test
```
**確認事項:**
- ノード数 (`n_nodes`) が安定しているか。
- K行列 (`K-field`) の成分が物理的な移動（距離の増減）に正しく反応しているか。
- 処理時間が 1〜3ms 以内に収まっているか。

---

## 3. 検証ステージの詳細

### **Stage 1: 距離感度テスト (Distance Sensitivity)**
- **内容**: 2物体が徐々に離れていく。
- **期待結果**: K行列の非対角成分 (K[0,1]) が指数関数的に減少すること。

### **Stage 2: 回転不変性テスト (Rotation Invariance)**
- **内容**: L字型構造が回転する。
- **期待結果**: 正準化 (CDU) により、回転中も K行列の構造が一定に保たれること。

### **Stage 3: 構造複雑性テスト (Multi-node Complexity)**
- **内容**: 十字型構造の移動。
- **期待結果**: 交差ノード (CROSS) を含む複雑なトポロジーにおいても、PKGF が安定して収束すること。

### **Stage 4: 照明耐性テスト (Real-world Synthesis)**
- **内容**: 陰影の変化を伴う球体の移動。
- **期待結果**: Otsu 二値化とトポロジー抽出が、動的な影の影響を排除して構造を捉え続けること。

---

## 4. ネイティブアプリの検証 (macOS)
ネイティブ版については、ビルド後に実機でカメラ映像との整合性を確認する。
```bash
./Scripts/build_macos.sh
./build_macos/PoISLAMApp
```
**確認事項:**
- Metal による 3D Grid が、カメラ映像の構造に応じてリアルタイムに変形しているか。
- ヴォクセルマップが累積され、視覚的に妥当な 3D 空間が構築されているか。
